# Product Service ‚Äî Spring Boot Backend

Reactive REST API for the Product Service platform, built with **Spring Boot WebFlux** following **Hexagonal Architecture** (Ports & Adapters). Provides CRUD operations for product management, backed by PostgreSQL (R2DBC), with Kafka event publishing, Redis-backed rate limiting, and JWT RS256 authentication.

Documentation is auto-generated by **Springdoc OpenAPI 2** from `@Operation`, `@ApiResponse`, and `@Schema` annotations on `ProductController`.

| Endpoint | URL | Notes |
|---|---|---|
| Swagger UI | `http://localhost:8080/swagger-ui.html` | Public ‚Äî no token required to view |
| OpenAPI spec (JSON) | `http://localhost:8080/v3/api-docs` | Public |

**To test authenticated endpoints from the Swagger UI:**

1. Generate a token ‚Äî inject `JwtService` and call `generateToken("user", "ADMIN")` (or use the Postman collection, which sets `{{adminToken}}` automatically).
2. Click **Authorize** in the Swagger UI and enter `Bearer <token>`.

Write endpoints (`POST`, `PUT`, `DELETE`) require `ROLE_ADMIN`. Read endpoints require any authenticated user.

## Security

The API is secured with **JWT RS256** tokens. A local RSA 2048 key pair (stored in `src/main/resources/certs/`) signs and verifies tokens.

| Route | Method | Required role |
|---|---|---|
| `/api/v1/**` | `GET` | Any authenticated user (`USER` or `ADMIN`) |
| `/api/v1/**` | `POST`, `PUT`, `DELETE` | `ROLE_ADMIN` only |
| `/actuator/**`, `/swagger-ui/**`, `/v3/api-docs/**` | Any | Public (no token needed) |

Token generation is handled by `JwtService` (available in the Spring context). For local testing, generate a token with:

```java
// inject JwtService and call:
String adminToken = jwtService.generateToken("alice", "ADMIN");
String userToken  = jwtService.generateToken("bob",   "USER");
```

Pass the token in the `Authorization` header:
```
Authorization: Bearer <token>
```

The Postman collection uses a `{{adminToken}}` environment variable ‚Äî set it in the active environment before running write requests.

## Database Migrations (Flyway)

Schema is managed by **Flyway** ‚Äî versioned SQL files in `src/main/resources/db/migration/` run automatically on startup.

```
db/
‚îú‚îÄ‚îÄ migration/           Applied in all environments (dev, prod, test)
‚îÇ   ‚îú‚îÄ‚îÄ V1__create_product_table.sql
‚îÇ   ‚îî‚îÄ‚îÄ V2__add_created_at_to_product.sql
‚îî‚îÄ‚îÄ testdata/            Applied in dev only (seed data)
    ‚îî‚îÄ‚îÄ R__seed_products.sql
```

| Migration | Description |
|---|---|
| `V1__create_product_table.sql` | Creates `product` table with constraints and index |
| `V2__add_created_at_to_product.sql` | Adds `created_at` timestamp column (schema evolution) |
| `R__seed_products.sql` | Repeatable ‚Äî inserts 3 sample products (dev only) |

Flyway uses a JDBC `DataSource` (HikariCP) running alongside the reactive R2DBC connection ‚Äî a common pattern for schema management in WebFlux applications. The `flyway_schema_history` table tracks applied migrations.

## Caching

`ProductPersistenceAdapter` implements cache-aside reads on top of Redis (`ReactiveStringRedisTemplate`, JSON via Jackson):

| Cache key | Populated by | TTL |
|---|---|---|
| `product-cache:{id}` | `findById` | 5 min |
| `product-sku-cache:{sku}` | `findBySku` | 5 min |
| `products-active-cache:{page}:{size}` | `findAllActive` | 5 min |

Both are invalidated (`KEYS` + `DEL` on their prefix) on every `save`/`update`/`delete`. This is a real distributed cache, not decoration ‚Äî with 2 replicas (`deployment.yaml`), it's shared across pods instead of each instance holding its own stale copy.

**Fail-open:** any Redis error (read, write, or invalidation) is logged as a warning and treated as a cache miss/no-op ‚Äî `ProductPersistenceAdapter` always falls back to PostgreSQL. Redis is not part of the Actuator readiness probe; if it's down, the pod stays `Ready` and keeps serving from Postgres, just without the cache speedup.

## Observability

The API exposes metrics at `/actuator/prometheus` (Micrometer + Prometheus registry) and distributed traces via OpenTelemetry (OTLP exporter, configurable via `OTEL_EXPORTER_OTLP_ENDPOINT`). All requests are logged with an `X-Request-Id` correlation header.

> **Design note:** `/actuator/prometheus` and `/swagger-ui.html`/`/v3/api-docs` are intentionally `permitAll()` (`SecurityConfig.java`) and reachable through the public Ingress ‚Äî the Postman collection's "Observability" folder exercises `/actuator/prometheus` directly against the k8s environment as part of the demo. Neither leaks application data: the actuator surface is metrics-only (no `env`/`heapdump`/etc. exposed), and Swagger only exposes the API *shape*, since every `/api/v1/**` call still requires a valid JWT. This is a deliberate portfolio tradeoff ‚Äî public docs/metrics to showcase the API, not an oversight.

### Structured JSON logging

In the `prod` profile, logs are emitted as **Elastic Common Schema (ECS)** JSON to stdout, ready for ingestion by Loki, Elasticsearch, or any log aggregator. `trace.id`/`span.id` are injected by Micrometer Tracing / OpenTelemetry; `requestId` is emitted by `RequestLoggingFilter`. In the `dev` profile, the default human-readable console format is used.

### Alerting

`chart/templates/prometheus-rule.yaml` defines a `PrometheusRule` (requires [Prometheus Operator](https://github.com/prometheus-operator/prometheus-operator)) with three rules: `HighErrorRate` (critical, >5% 5xx for 2 min), `HighP99Latency` (warning, P99 >1s for 2 min), `PodNotReady` (critical, any pod not ready for 2 min).
